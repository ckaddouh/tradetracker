**Q1 — Scope decisions**
Compare your original BRIEF.md feature list to what you actually built. What did you cut, add, or change, and why?
In my original brief, I aimed to have the following:
- A page to view your journal entries (which will include a stock name, position, reasoning, and date)
- A page to view the performance of your ideas over time
- A page to view upcoming earnings reports to be able to have this information all in one place. 

I ended up implementing the first and third features, but decided against the second because I realized it made my tool behave as a investment "simulator" of sorts, which is not what I was aiming for. Instead, in the journal page, I allow users to see if their trades are earning a profit or loss as time goes on without having a whole dashboard to describe it. I did add a much larger feature which can be seen as the "Analyzer" tab. I got inspired from this idea because I realized a computer can go through the same thought process that I go through while reading news articles, only faster and better. This feature can take a company, read its most recent annual filing, and break down its critical revenue streams into smaller and smaller sub-factors. Then, you can input a news article to see which companies it might affect based on their trees of supply chains. 


**Q2 — Technical challenge**
Describe the single hardest technical problem you encountered. What was the error or obstacle, and how did you resolve it?
- The hardest technical problem I encountered was probably figuring a way around API limits for the AIs that I am using within the web app. I use the free tier of Groq AI for both the earnings summaries page (for the AI summary of the earnings report) as well as for the analyzer page to create each company's supply chain tree and then analyze a news article to see which companies it impacts. To get around this, I tried using Gemini for the latter, but found that its free tier has a limit of 0 in the EU. I will probably resolve this in the future by putting in billing information, but for now, I am planning to use Groq. I lowered the max token limits on its responses and also shortened my prompts and cached some of the context to decrease the amount of tokens used in each prompt. Still, I will likely have to input billing information in the future if I would like to scale the analyzer feature.

**Q4 — Architecture**
Explain how data flows through your application for one specific user action — from the moment the user clicks something, through the front-end, the API, the database, and back to the screen.
- A specific user action I will explore is a jounral entry in the first tab. It begins from a user clicking on "New Idea" in the Jounral tab. Then, they fill out the form which is created in JournalView.vue, and click "Save Idea." Before anything is saved, the frontend first validates the ticker by hitting /api/trades/chart/:ticker. The backend then sends this to Yahoo Finance to ensure that it is a valid ticker (if not, it sends an error to the user and they must try again). Once their form submission is finalized, handleSubmit calls tradeStore.createTrade(form.value). The store sends a POST request to /api/trades with an Authorization: Bearer <token> header attached via getHeaders(). On the server, Express routes this to trades.js, where the verifyToken middleware runs first (to decodes the JWT and attaches the user's ID to req.user). The route handler then calls TradeIdea.create(...), which persists the document to MongoDB. Then, the saved document is returned in the response. Back in the store, this.trades.unshift(res.data) prepends it to the local array so that the new card appears at the top of the list immediately, without needing a full refetch.

- Next, when the user clicks a TradeCard, openChart() fires and sets showModal = true, which renders the modal via Vue's <Teleport> into the document body. loadChart() is then called immediately, so then when the user wants to view the stock's history, they can see it with minimal lag time. If the trade is closed and has a closedAt date, it also finds the exit price and computes a separate closed P&L. Finally, renderChart dynamically imports Chart.js and draws the canvas, coloring the line segment before the idea date in a muted grey and the segment after in green or red depending on P&L. A custom ideaLine plugin draws the dashed vertical markers on top.

- Finally, if a user clicks "Delete" on a card, the card emits a delete event with trade._id up to JournalView, which calls tradeStore.deleteTrade(id). The store sends a DELETE request to /api/trades/:id. The backend uses findOneAndDelete with both _id and userId as conditions (to prevent one user from deleting another's trade ideas somehow). The store then filters the deleted ID out of this.trades locally, so the card disappears from the UI instantly without waiting for a refetch again.
